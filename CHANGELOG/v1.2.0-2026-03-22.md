# Changelog — v1.2.0

**Released**: 2026-03-22
**Type**: Security patch + reliability improvements

---

## What Changed

### 🔐 Security Fixes

#### Webhook Signature Verification — Now Correct and Timing-Safe
Previous versions computed the HMAC signature over a re-serialized version of the request body, which could differ from the original bytes and cause valid requests to be rejected or invalid ones accepted.

This release:
- Uses the **raw request bytes** (as originally sent) for HMAC computation, matching the server-side signing behavior exactly
- Replaces the Web Crypto API (`crypto.subtle`) with Node.js built-in `createHmac` — more reliable across all n8n deployment environments
- Switches from plain string comparison to **constant-time comparison** (`timingSafeEqual`) to prevent timing-based signature enumeration attacks

#### Base URL Validation (SSRF Prevention)
The Base URL credential field now validates the URL before any HTTP request is made:
- Rejects non-HTTP/HTTPS protocols
- Blocks all private/internal network addresses (loopback, RFC 1918, link-local) using exact octet-boundary checks — not prefix strings that could match legitimate domain names
- Provides a clear error message if the URL is invalid

#### Resource ID Validation (Path Traversal Prevention)
All ID parameters (Lead ID, Client ID, Webhook ID) are now validated before being used in API request URLs:
- Empty strings throw a clear error instead of silently hitting the wrong endpoint
- IDs containing path-traversal characters (`/`, `..`, `?`) are rejected
- Error messages indicate exactly which field failed and why

---

### ⚡ Reliability Improvements

#### Request Timeout
All outbound API requests now have a **30-second timeout**. Previously, a slow or unresponsive API could cause n8n executions to hang indefinitely.

#### Error Surfacing for Dynamic Field Loader
When the node loads your custom lead fields from Brixfit, any API failure (bad credentials, network error) now surfaces a clear error message. Previously the node silently returned an empty field list, leaving you with no indication something was wrong.

#### JSON Parse Error Handling
If the Brixfit API ever returns a non-JSON response (e.g. due to a proxy or server error), the node now throws a descriptive `NodeOperationError` that includes the first 200 characters of the raw response — making diagnosis straightforward instead of crashing with a generic `SyntaxError`.

#### Zero-Value Filter Preservation
Query filters (page, per_page, numeric fields) no longer drop values of `0`. The previous filter stripped out `0` alongside empty strings, which could unintentionally suppress valid filter values.

#### Per-Item Headers
Request headers are now created fresh for each item in the execution loop, preventing any possible mutation bleed between items.

#### Item Correlation (`pairedItem`)
All output items now carry correct `pairedItem` references, enabling proper item tracing in multi-branch workflows and merge node operations.

---

### 🔧 UI / UX Fixes

#### Credential Connection Test
Clicking **"Save"** or **"Test connection"** on the Brixfit API credential now makes a live API call and shows an immediate success or error message — no more saving blind.

#### Client Check-ins Filter Options Now Visible
The **Client → Get Check-ins** operation now correctly shows its filter options (status, date range, page, per page) in the node UI. Previously these options were defined but never rendered due to an incorrect `displayOptions` scope.

#### Webhook `getAll` No Longer Shows Lead/Client Filters
The **Webhook → Get All** operation previously inherited filter inputs (search, status, sort) that only apply to leads and clients. These have been correctly scoped out so the webhook list view is clean.

#### Lead `getAll` Sort Option
The **Lead → Get All** operation now exposes a **Sort** dropdown (newest first, oldest first, name A–Z, name Z–A) that maps to the Brixfit API `sort` parameter.

#### Trigger Credentials Now Required
The Brixfit Trigger node previously marked credentials as optional. They are now correctly marked as **required** — the trigger cannot function without a valid API key.

---

## Upgrade Notes

- **No breaking changes** to workflow behavior for valid inputs
- Workflows using invalid or empty IDs in expressions will now surface a clear error instead of silently misbehaving — review any workflows that use dynamic ID expressions
- The Base URL field now rejects private/internal IPs — self-hosted instances must use a publicly reachable hostname
- If you use **Client → Get Check-ins**, open the node and re-add any filters you need (they were previously invisible and not being saved)
