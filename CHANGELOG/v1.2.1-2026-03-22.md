# Changelog — v1.2.1

**Released**: 2026-03-22
**Type**: Reliability patch + UI fixes

This patch completes the remaining fixes from the v1.2.0 security audit cycle.

---

## What Changed

### ⚡ Reliability Fixes

#### Dynamic Field Loader — Error Now Surfaced
When the node loads your custom lead fields from Brixfit, any API failure (bad credentials, network error, etc.) now throws a descriptive error immediately. Previously the node silently returned an empty field list, giving no indication that something was wrong.

#### Zero-Value Filter Preservation
Query filters with a value of `0` (e.g. `page: 0`, numeric filters) are no longer silently dropped. The previous filter stripped `0` alongside empty strings, which could suppress valid filter values sent from expressions.

#### JSON Parse Error Handling
If the Brixfit API returns a non-JSON response (e.g. due to a proxy or a server error), the node now throws a clear `NodeOperationError` that includes the first 200 characters of the raw response — making diagnosis easy instead of crashing with a generic `SyntaxError`.

---

### 🔧 UI / UX Fixes

#### Client → Get Check-ins Filters Now Visible
The **Client → Get Check-ins** operation now correctly renders its filter options (status, date range, page, per page) in the n8n node UI. Previously the options block existed in the node definition but was scoped incorrectly and never displayed.

#### Webhook → Get All No Longer Shows Irrelevant Filters
The **Webhook → Get All** operation previously inherited lead/client filter inputs (search, status, sort). These have been scoped out — the webhook list view is now clean with no unnecessary filter fields.

#### Lead → Get All Sort Option Exposed
The **Lead → Get All** operation now exposes a **Sort** dropdown (Newest First, Oldest First, Name A–Z, Name Z–A) mapped to the Brixfit API `sort` parameter.

#### Brixfit Trigger — Credentials Now Required
The Brixfit Trigger node previously marked credentials as optional (`required: false`). They are now correctly marked as **required** — the trigger cannot authenticate or register webhooks without a valid API key.

---

## Upgrade Notes

- No breaking changes
- If you use **Client → Get Check-ins**, open the node and re-configure any filters (they were previously invisible and not being saved)
- Workflows relying on filter value `0` being silently ignored may behave differently — verify any dynamic filter expressions
